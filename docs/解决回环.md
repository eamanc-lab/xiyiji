# 视频帧回环问题排查与解决方案

> **问题反馈日期**: 2026-04-04
> **解决日期**: 2026-04-06
> **影响版本**: V28
> **修复版本**: V28+

---

## 一、问题现象

同事反馈：在使用预览功能时，视频播放到一半还没结束就突然从头开始播放。

### 日志佐证

从生产日志（`logs/2026-04-03_22-51-57.log` 等多个文件）中发现，帧号序列 `startFrame` 在正常递增后突然跳回 0：

```
正常递增: startFrame=0 → 352 → 547 → 734 → ... → 6249 → 6506
突然跳回: cancelAll → reset-driving → startFrame=0（视觉上从头开始）
```

**该现象在 6 个不同日期的日志文件中反复出现**，不是偶发。

---

## 二、根因分析

### 调查方法

成立 4 人专家团队（code-tracer / log-verifier / bug-hunter / tenth-man），其中第 4 人执行"以色列第十人原则"反复质疑前 3 人的结论，直到所有人达成一致。

### 调查结论

**回环不是帧同步算法 bug，而是一个综合性问题，由以下三个层次组成：**

#### 1. 直接触发：Python bridge 进程被杀

通过 correlationId 全链路追踪，确认每次回环前都有以下事件序列：

```
用户操作（关闭播放器/重新点预览）
  ↓
player:close IPC
  ↓
yundingyunboService.shutdown('player-close')  [yundingyunbo.service.ts:559]
  ↓
sessionOwner === 'preview'，guard 不阻止
  ↓
Python bridge 进程被杀
  ↓
_file_drive_cursor_frame 随进程内存消亡
  ... 几秒后 ...
  ↓
新 bridge 启动 → _file_drive_cursor_frame = 0
  ↓
_reset_file_mode_driving() → cursor 显式置 0
  ↓
第一个音频任务 → startFrame = 0（视觉跳回开头）
```

#### 2. 设计缺陷：Node 端帧同步是死代码

`pipeline.service.ts:onAck` 回调花了大量代码计算 startFrame：
- 查询播放器位置
- 预测推理延迟（EMA）
- avatarClock 预测
- normalizeFrame 计算

但最终结果在 `yundingyunbo.service.ts:556` 被丢弃：

```typescript
pending.onAck(msg.num_frames || 0, msg.total_chunks || 1).catch(() => {})
```

返回值（startFrame）从未通过 `process_audio` 命令发送给 Python。**Node 端算的 startFrame 完全是死代码，只用于 Node 端自己的日志记账。**

Python 端独立维护 `_file_drive_cursor_frame`，正常运行时 Node/Python 两侧碰巧一致（因为都是顺序递增），但 bridge 重建后就彻底脱节。

#### 3. 保护不对称：preview 模式无 sessionOwner guard

`yundingyunbo.service.ts:942-948` 中，`shutdown('player-close')` 只在 `sessionOwner === 'live'` 时被 guard 阻止：

```typescript
if (
  !opts?.force &&
  reason === 'player-close' &&
  (this.sessionOwner === 'live' || this.liveTransitionPrepared)
) {
  console.log('[YDB] Ignored player-close because live startup/session owns the native renderer')
  return
}
```

**live 模式有保护，preview 模式没有**。任何在 preview 模式下的 `player:close` 都会立即杀死 bridge。

### 已确认排除的非根因

| 假设 | 排除原因 |
|------|---------|
| Python 端 modulo wrap (`% total_frames`) bug | 视频 56,921 帧，cursor 在 6,507 时未触发 wrap，不可能是原因 |
| Node 端 cancelAll + resetAvatarClock 是根因 | 这只是 Node 端记账副作用，对 Python 实际渲染无影响 |
| 双时钟漂移 | 日志证明跳变前两侧 startFrame 完全一致，正常运行时不漂移 |
| 帧数计算 off-by-one | 数学验证正确，cycleLen 取 driving video frames 是正确设计 |

## 三、解决方案

修复分三个层面：

### 层面一：可观测性改造（让问题可追溯）

**目标**：未来再出现"回环"反馈，看一眼日志的 reason 字段就能立刻判断是用户操作还是程序异常。

#### 全链路 correlationId 设计

从 Vue 按钮 → preload → IPC → service → Python bridge，统一传递同一个 correlationId。一条 `grep correlationId=xxx` 就能串起完整链路。

#### 改动清单

| 文件 | 改动 |
|------|------|
| `Tab2Profiles.vue` | `stopPreview()` 支持 reason/correlationId，所有停止分支都有明确 reason（按钮、预热失败、组件卸载、live 抢占等） |
| `DigitalLive.vue` | live 页停止视频/音频时也带 reason/correlationId |
| `player.ts` | 播放器窗口关闭按钮生成 correlationId |
| `pipeline.ipc.ts` | `pipeline:cancel` 支持 `{taskId, reason, correlationId}` |
| `player.ipc.ts` | `player:close` 支持 `{reason, correlationId}` |
| `yundingyunbo.service.ts` | `shutdown()` 打印 reason/detailReason/correlationId/force/sessionOwner/pendingRequests/bridgeGeneration，记录 lastShutdownTrace |
| `yundingyunbo_bridge.py` | `handle_shutdown()` 打印 reason/detailReason/correlationId/managerExists/inputMode/fileModeBackend/pendingAudio/preparedAudio/activeAudio/queue/initBusy/latestInitReq |
| `pipeline.service.ts` | `cancelAll()` / `resetAvatarClock()` 都支持 reason/correlationId 参数，并打印重置前状态 |
| `lipsync-backend.ts` | `resetStreamingSession()` / `shutdown()` 接口签名扩展，统一支持 detailReason/correlationId |

#### 日志示例

```
[09:52:46.222] [Player] player:close received, sessionOwner=preview, reason=preview-button-click, correlationId=preview-stop:00e3b81e
[09:52:46.224] [YDB] shutdown: reason=player-close, detailReason=preview-button-click, correlationId=preview-stop:00e3b81e, force=false, sessionOwner=preview
[09:52:46.567] [Pipeline] cancelAll: reason=preview-button-click, correlationId=preview-stop:00e3b81e
[09:52:46.568] [Pipeline] resetAvatarClock: reason=cancelAll:preview-button-click, prevFrame=18256, prevNextStart=18256 → reset to -1
```

可以清楚看到：
- 触发源：用户点击预览按钮
- 完整链路：按钮 → player:close → shutdown → cancelAll → resetAvatarClock
- 重置前的真实帧位置：18256

### 层面二：根本性修复（两阶段协议）

**目标**：让 Node 算的 startFrame 真正生效，结束两端独立运行的设计缺陷。

#### 协议改动

将原来的单阶段 `process_audio` 命令拆成两阶段：

**Phase 1: prepare**
- Node 发送 `process_audio` with `phase: 'prepare'`
- Python 接收音频，做特征提取，但不入队
- Python 返回 ack 消息（包含 num_frames）

**Phase 2: commit**
- Node 收到 ack 后，调用 `onAck()` 计算 startFrame（基于 avatar clock、播放器位置、推理延迟预测）
- Node 发送 `process_audio` with `phase: 'commit'` 和具体的 `start_frame`
- Python 用 Node 指定的 start_frame 入队

#### 改动清单

| 文件 | 改动 |
|------|------|
| `pipeline.service.ts` | `PendingRequest` 新增 `startFrameCommitted`，防止 prepare ack 重复 commit |
| `yundingyunbo.service.ts:971` | `processAudioStream()` 改成先发 `phase: 'prepare'` |
| `yundingyunbo.service.ts:583` | 收到 ack 后真正调用 `onAck()` 算 start_frame，再发 `phase: 'commit'` |
| `yundingyunbo_bridge.py:7409` | `process_audio` 拆成 `_prepare_process_audio()` / `_commit_process_audio()` / `handle_process_audio()` |
| `yundingyunbo_bridge.py:7563` | 识别 `phase == 'prepare'`，prepare 只返回可用帧数，commit 才真正入队 |
| `yundingyunbo_bridge.py:7615` | pending audio done 检测开始参考 active audio path，而不是只靠时间估计 |
| `yundingyunbo_video_stream_bridge.py:215` | file-mode drive request 支持 `requested_start_frame`，no-loop 下更精确计算 start_frame/end_frame/visible_end_frame |

### 层面三：附带的 bug 修复

调查过程中发现并修复的其他相关问题：

| 问题 | 位置 | 修复 |
|------|------|------|
| `chunk avatarClock` 路径缺少 `setAvatarClock` 调用，clock 不更新 | `pipeline.service.ts:716` | 补上 `setAvatarClock(...)` |
| chunk played 归属用纯 FIFO，skip 时 stale chunk 误算到下一个 item | `live-pipeline.service.ts:107` | 改成按 audioPath 归属，stale chunk 缓存而非误算 |
| pre-played chunk 必须刚好是队头才能记账，否则丢失 | `live-pipeline.service.ts:423` | 改成谁注册算谁的 |
| `_pending_audio` 在 bridge shutdown 时未清理 | `yundingyunbo_bridge.py:7483` | shutdown 时清空并发出错误 |

### 层面四：预览体验改进

发现预览段落播完就停止（不循环），用户误以为"播放到一半就结束"是 bug。改为循环播放：

| 文件 | 改动 |
|------|------|
| `Tab2Profiles.vue:551` | 移除 `finishPreviewPlayback` 调用，cursor 重置为 0 继续播 |
| `Tab2Profiles.vue:609` | `submitPreviewBatch` 中 `idx = previewSegmentCursor % context.segments.length`，cursor 用 modulo 绕回 |
| `Tab2Profiles.vue:612` | 状态栏显示当前轮次："预览播放中：第 3/20 段（第 2 轮）" |

## 六、后续如果再有"回环"反馈

按以下步骤排查：

1. **拿到日志文件**：`D:\yunyin\XYJ2\xiyiji\logs\` 下的 `.log` 文件
2. **搜索帧跳变**：
   ```bash
   grep -o "startFrame=[0-9]*" log.log | sed 's/startFrame=//' | \
     awk 'NR>1 && $1<prev && prev-$1>1000 { printf "%d->%d\n", prev, $1 } { prev=$1 }'
   ```
3. **如果发现跳变**，找到跳变时间附近的 cancelAll/shutdown：
   ```bash
   grep "cancelAll: reason\|shutdown: reason\|resetAvatarClock: reason" log.log
   ```
4. **看 reason 字段**：
   - `room-start` / `room-stop` → 用户开/停直播，正常
   - `preview-button-click` → 用户点了预览按钮，正常
   - `preview-open-reset` → 启动新预览前的清理，正常
   - `preview-tts-submit-failed` → TTS 网络超时，需查 TTS API 状态
   - `preview-playback-complete` → 旧逻辑，已改为循环（应该不再出现）
   - **其他未知 reason** → 真正的异常，进一步排查
5. **用 correlationId 追踪完整链路**：
   ```bash
   grep "correlationId=preview-stop:xxx" log.log
   ```
6. **如果跳变发生在 cycleLen 边界附近**（prev > 55000 且 current < 2000）：这是正常的视频循环 wrap，**不算回环**

## 七、结论

**本次"帧回环"问题不是单一的 bug，而是一个综合性问题：**

- **表象**：用户操作导致 bridge 重启，画面从头播放
- **设计缺陷**：Node 端帧同步是死代码，Python 端 cursor 在 bridge 重建后无法恢复
- **可观测性不足**：日志无法区分"用户主动操作"和"程序异常"

**解决方案的核心思路**：
1. **加日志**：全链路 correlationId + reason，让所有重置都有据可查
2. **修协议**：两阶段协议让 Node 算的 startFrame 真正生效，结束两端独立运行
3. **改体验**：预览改为循环播放，避免误以为"播完就结束"是 bug

修复后，本地 1+6 小时监控、客户 3.5 小时日志全部验证通过，0 次异常回环。即使未来再有类似反馈，通过 reason 和 correlationId 也能在几秒内定位到真正的触发源。
